Pi Audio Feedback Release Guide

This guide publishes pi-audio-feedback to npm and creates the matching GitHub Release.

One-time setup

Confirm Node.js

node --version
npm --version

Use Node.js >=22.19.0.

Authenticate with npm and GitHub

npm login
npm whoami
gh auth status

First release: v0.1.0

Run all commands from the repository root.

1. Review changes

git status
git diff

Confirm that all intended changes are present.

2. Run release checks

npm ci
npm run lint
npm run format:check
npm run typecheck
npm test
npm run assets:verify
npm pack --dry-run --json
npm audit --omit=dev --audit-level=high

Do not continue if any check fails.

3. Inspect package contents

npm pack --dry-run

Confirm that runtime source, WAV assets, LICENSE, README.md, package.json, and the PowerShell helper are included. Tests and development files should be excluded.

4. Commit and push

git add .
git commit -m "Prepare release v0.1.0"
git push origin main

5. Publish to npm

This is an unscoped package, so it is public by default.

npm publish

If two-factor authentication is requested:

npm publish --otp=123456

Verify publication:

npm view pi-audio-feedback version

6. Create the GitHub Release

gh release create v0.1.0 --generate-notes --title "v0.1.0"
gh release view v0.1.0

Future releases

Use the appropriate version bump:

npm version patch   # 0.1.0 -> 0.1.1
npm version minor   # 0.1.0 -> 0.2.0
npm version major   # 0.1.0 -> 1.0.0

npm version updates package.json and package-lock.json, then creates a Git commit and tag.

After testing the new version:

npm test
npm run assets:verify
git push origin main --follow-tags
npm publish
gh release create vX.Y.Z --generate-notes --title "vX.Y.Z"

Release rules

  • The version must match in package.json, npm, the Git tag, and the GitHub Release.
  • A version already published to npm cannot normally be published again.
  • Never publish until automated checks and required manual acoustic checks pass.
